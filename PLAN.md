# yw-look プラン

## 1. 製品の定義

### 目指すもの

`yw-look` は、Windows 上で DCC / CG 制作現場のアセットを「雑に・速く・気持ちよく確認する」ための軽量ビューアである。

これは 3D ソフトやアセット管理ツールではなく、`macOS Preview / Quick Look` に近い体験を CG アセット向けに再構成した **Asset Quick Look** である。

### 解決したい課題

Windows では以下のような実務ファイル群を、軽快な UI で即座に開いて確認できる統合ビューアが弱い。

- `glTF / GLB`
- `FBX`
- `OBJ`
- `USD / USDZ / USDA / USDC`
- `DDS`
- `HDR`
- `EXR`
- `TGA`

既存ツールは次のどれかに寄りがちで、実務上の「ちょっと見たい」を満たしにくい。

- UI が古い
- 起動や操作が重い
- 対応フォーマットが偏る
- USD の実務上の扱いが弱い
- テクスチャ単体確認に弱い
- DCC ワークフローの「雑に開く」に向いていない

### プロダクトの価値

価値の中心は「高機能」ではなく「確認の速さ」である。

- すぐ起動する
- D&D ですぐ見られる
- 右クリックや関連付けから開ける
- モデルもテクスチャも一つのアプリで確認できる
- 破綻していても状況把握がしやすい

### 参考にする体験

目指す体験の基準は、`macOS Preview / Quick Look` と `XnView` の中間にある。

- `macOS Preview / Quick Look` のように、余計な準備なしで素早く開ける
- `XnView` のように、雑多なファイルを軽快に順送りで確認できる
- そのうえで 3D モデル確認に必要な最低限の操作を持つ

言い換えると、「`XnView` に 3D モデルビューアが自然に載っている状態」に近い感覚を目指す。

ユーザーが「確認のために DCC を起動しなくてよくなった」と感じること。

## 7. フォーマット対応方針

### glTF / GLB

最重要対応対象。MVP で安定動作させる。

### FBX

実務上の需要が高いので優先対応。ただし複雑なケースでは見え方の差が出る可能性を許容する。

### OBJ

実装コストが低く、最低限の確認用途に有効。

### EXR / HDR

プレビュー需要が高い。巨大ファイルではメモリ負荷が高いため、将来的には縮小版生成や表示解像度制限を検討する。

### TGA

優先度高。比較的扱いやすいので MVP 対応する。

### DDS

MVP で対応対象に含める。ただし GPU 圧縮や環境差の問題があるため、初期実装では「表示できる範囲で扱う」方針にする。必要なら Rust 側で RGBA 展開を検討する。

### USD

完全対応は狙わない。初期は Three.js 系の既存 USD ローダーを前提にし、「単純なケースが見えたら十分」と定義する。参照解決や payload、variant などの複雑性は段階導入する。

### USD の最低対応ライン

初期段階では、Three.js 系の既存 USD ローダーで読み込める範囲をそのまま最低ラインとする。

- 独自の完全対応は目指さない
- 複雑な参照や production-ready な互換性は約束しない
- 「開けたら助かる preview support」として扱う

## 9. マテリアル / 描画方針

初期段階では、マテリアル再現性そのものを競争力にしない。

この製品は DCC 間の完全互換ビューアではなく、アセット確認のための quick look である。そのため、MVP では「完全再現」より「大きく破綻せず、確認用途に十分な見え方」を優先する。

### 基本方針

- 描画の基準は `Three.js` の PBR 系に置く
- 初期は `MeshStandardMaterial` / `MeshPhysicalMaterial` 系を前提にする
- `glTF / GLB` の PBR を最も自然に扱える状態を基準にする
- 他形式は、可能な範囲で標準 PBR に寄せて吸収する

### 互換性の考え方

- `glTF / GLB` は最も相性が良い基準フォーマットとして扱う
- `FBX` や `OBJ` は完全再現ではなく preview 重視で扱う
- `USD` はさらに preview support 色が強いものとして扱う
- DCC / renderer ごとの差異は最初から残る前提で設計する

### 初期段階で目指す見え方

- ベースカラーが大きく崩れない
- 透明やカットアウトが致命的に壊れない
- 法線が大きく破綻しない
- 金属感 / 粗さ感が概ね妥当に見える
- 環境マップ込みで「確認には十分」と言える

### 初期段階で割り切ること

- DCC ごとの独自シェーダ再現はしない
- Arnold / V-Ray / Redshift / Blender 独自ノード互換はしない
- Adobe 系や Khronos 系の差異を完全吸収することは目指さない
- 複雑なノードグラフは扱わない

### 実装上の整理

- `glTF` では loader が持つ PBR 情報をできるだけそのまま活かす
- `FBX` は loader から得られる情報を `Three.js` の標準 PBR マテリアルへ寄せる
- `OBJ` は最小限の見え方を優先する
- `USD` は既存ローダーが返す範囲で扱う

### 見た目補助として必要なもの

- 環境マップ切り替え
- 背景切り替え
- ワイヤーフレーム切り替え
- 必要なら法線確認系の簡易表示

### 初期の表示モード

初期段階のマテリアル / 表示確認モードは以下の 4 パターンを基本とする。

- テクスチャなし
- テクスチャあり
- ワイヤーフレーム
- テクスチャあり + ワイヤーフレーム重ね表示

### 将来的な拡張余地

- マテリアルの簡易インスペクト表示
- テクスチャスロット確認
- 法線 / Roughness / Metallic などの確認モード
- 形式ごとの変換ルール追加

### 使用テクスチャ確認の方針

3D モデルに紐づくテクスチャを一覧で見られる機能は、確認ツールとして価値が高い。

- モデル内で参照されているテクスチャを列挙する
- 埋め込みテクスチャがある場合は個別にプレビューできるようにする
- 外部参照テクスチャも、解決できたものは一覧に含める
- 初期段階では編集や差し替えは行わず、閲覧専用とする

### 初期段階で見たい情報

- テクスチャ名
- 種別の推定
- 埋め込みか外部参照か
- 解決済みか未解決か
- 個別プレビュー

## 10. パフォーマンス方針

軽さは技術選定より実装方針で決まる。初期から以下を徹底する。

- 拡張子ごとの loader を lazy load
- ファイル切り替え時の `dispose` を徹底
- `geometry / material / texture` の明示解放
- 巨大テクスチャの扱いを制限
- EXR / HDR など重いファイルの表示負荷を意識する
- 初期は一覧ビューや一括読み込みを入れない

### パフォーマンスの最低要件

- 小〜中規模ファイルはストレスなく開ける
- 読み込み失敗でアプリ全体が不安定にならない
- 連続でファイルを開いてもメモリが積み上がりにくい

## 11. UX / UI 方針

### UI の方向性

- 今っぽいが軽い
- 情報密度は抑える
- まず表示面積を優先する
- 初見で迷わない
- 画像ビューアに近い静かな UI をベースにする

### UI の基本思想

UI は「超シンプルを基本にして、3D に必要な要素だけを薄く乗せる」。

- 常時 UI は最小限に抑える
- 3D 特有の操作は必要なときだけ見せる
- サイドバー常設は初期段階では避ける
- ビューアの表示面積を最優先にする

### UI の基本構成案

- 中央: ビューア
- 上部: 最小ツールバー
- 下部または右下: ファイル名、簡易ステータス
- 必要時のみ表示されるオーバーレイ

### 常時表示する UI 候補

- `Open`
- `Prev`
- `Next`
- `Wire`
- `Env`
- `Bg`

### 条件付きで表示する UI 候補

3D モデルにアニメーションがあるときだけ、動画プレイヤーに近いミニ操作バーを表示する。

- `Play / Pause`
- シークバー
- フレーム送り / 戻し
- 時間表示
- アニメーションクリップ選択

### アニメーションバーの配置方針

- プレビュー画面の下部にオーバーレイ表示する
- `Blender` や一般的な動画プレイヤーに近い見た目でよい
- アニメーションがあるモデルのときだけ表示する
- アニメーションがない場合は非表示にする

### 初期 UI で避けるもの

- 常設の大きなサイドバー
- 複雑なインスペクタ
- 多段ツールバー
- 編集系 UI
- DCC 的なパネル構成

### 3D 操作系の方針

カメラ操作は、一般的な 3DCG ツール利用者にとって違和感の少ない系統を採用する。

- `Unity` や `Maya` などで馴染みのある挙動を基準にする
- 実装の土台は `Three.js` の `OrbitControls` 系を使う想定とする
- 初期段階では、独自の複雑なカメラシステムは作らない

### 初期のカメラ操作イメージ

- Orbit
- Pan
- Zoom

具体的な割り当ては、一般的な DCC の感覚に近づける。

### 操作感の優先順位

- 初見で壊れて見えないこと
- DCC ユーザーが直感的に触れること
- 実装コストが過剰に膨らまないこと

### 初期実装での割り切り

- まずは `OrbitControls` ベースで成立させる
- 必要なら後から Maya 寄りのショートカットや操作プリセットを検討する
- 初期段階では複数操作モードの併存は避ける

### カメラ初期位置

- 読み込み時は対象全体が見えるように `fit to view` を行う
- 初期視点は斜め上からの見下ろし寄りにする
- 対象のスケールが極端に小さい / 大きい場合は警告を出す

### up-axis 方針

- ビューア内部の基準は `Y-up` とする
- `Unity` 系に近い感覚を優先する
- `Z-up` 系データとの差異は初期段階では許容し、必要なら警告や補助表示で扱う

### 初期のカメラ割り当て方針

Maya 系の操作感を優先する。

- `Alt + 左ドラッグ`: Orbit
- `Alt + 中ドラッグ`: Pan
- `Alt + 右ドラッグ`: Zoom

必要ならホイールズームも併用する。

### 背景と環境の初期方針

- 背景は灰色ベースにする
- 初期段階ではグラデーション寄りの落ち着いた見え方を優先する
- 環境マップはまず 1 個だけ持つ
- 背景や環境の切り替え拡張は後から追加できる前提にする

### テクスチャ単体ビューの初期方針

- `RGB`
- `RGBA`
- `Alpha`

のチャンネル切り替えを持たせる。

- `HDR / EXR` ではトーンマップ調整を持たせる
- 白飛びや黒つぶれを確認しやすい方向にする
- `DJV` のような実務向け EXR ビューアの最低限の気持ちよさを参考にする

### 表示したい最小情報

- ファイル名
- 拡張子
- 読み込み成功 / 警告 / 失敗
- 必要なら簡易統計情報

### メタデータ表示の初期方針

初期段階では、確認用途に効く基本情報を優先して表示する。

- ファイル形式
- 形式バージョン
- ノード数やメッシュ数などの簡易統計
- アニメーション有無
- 使用テクスチャ数

`glTF` では、バージョン情報など取りやすいものを優先する。

### 階層構造表示の方針

- 3D モデルでは階層構造の確認価値が高いため、早い段階で対応候補に入れる
- 初期から読み取り専用の簡易ツリー表示を入れる
- 編集や選択同期のような高度機能は初期段階では持たない
- `USD` も、まずはテキストプレビューより階層確認を優先する

### エラー詳細の粒度

- 画面上は簡潔なユーザー向けエラーを出す
- 内部的にはもう少し具体的な理由を保持する
- 初期段階では開発者向けの詳細ダンプ UI までは持たない

最低限区別したいカテゴリ:

- 未対応形式
- 読み込み失敗
- 参照解決失敗
- テクスチャ欠損
- スケール警告

### 後で検討

- 右クリックから開く導線
- エクスプローラー連携の強化
- Quick Look ライクなショートカット起動
- サムネイルやプレビュー拡張

## 13. 起動速度と軽量性の考え方

この製品では、機能数よりも「軽く感じること」が重要である。

### 基本認識

- `Tauri` を使っても自動的に爆速になるわけではない
- 画像より 3D モデルの方が重くなりやすいのは前提とする
- 体感速度は、技術選定より初期設計で大きく決まる

### 起動を軽く保つ方針

- 起動時は viewer の骨格だけ出す
- 各フォーマット loader は遅延読込する
- 起動時に重い初期化をしない
- 最近使ったファイルや設定の読み込みは最小限にする
- 一覧機能や管理機能を初期から背負わない

### 目指す体感

- まずアプリがすぐ立ち上がる
- 重いファイルは開くときにだけ待つ
- 失敗しても次の確認にすぐ移れる
- 連続で開いても腰が重くなりにくい

### 初期キャッシュ方針

- MVP ではキャッシュを持たない
- サムネイルキャッシュも初期段階では持たない
- 必要性が高まったら後から個別に導入を検討する

### 計測基準

体感ではなく実測で詰める。最低限、以下を計測対象にする。

- 起動時間
- 初回表示時間
- 前後移動時の切り替え時間

可能な限りミリ秒単位で計測し、改善対象として扱う

## 14. フォルダ内前後移動の仕様方針

フォルダ内前後移動は MVP に含める。

### 基本挙動

- 現在開いているファイルが存在するフォルダを基準にする
- そのフォルダ内の「対応拡張子ファイル」を列挙する
- 列挙順に従って前のファイル / 次のファイルへ移動する
- 画像と 3D を区別せず、対応対象であれば同一の並びで扱う

### この方針の意味

- `png` の次が `fbx` でもよい
- `fbx` の次が `exr` でもよい
- ファイル確認をフォルダ単位で流す用途を優先する

### 初期実装での割り切り

- 並び順は OS 依存にせず、アプリ側で安定したルールを持つ
- 初期は単純なファイル名順を基本にする
- サブフォルダは対象外にする
- 非対応拡張子はスキップする
- 読み込み失敗時も前後移動は継続できるようにする

### ショートカット方針

- `Left Arrow`: 前のファイル
- `Right Arrow`: 次のファイル

## 15. エラーハンドリング方針

壊れたファイルや未対応ケースは前提として扱う。

### 必要な振る舞い

- 落ちない
- フリーズしにくい
- 何が起きたか最低限わかる
- 再読み込みしやすい

### エラーメッセージの方針

- 技術詳細を出しすぎない
- ただし調査に使える最低限の情報は保持する
- 「未対応」「参照解決失敗」「テクスチャ欠損」などを区別する

### 初期のエラー表示方針

- 読み込み失敗時はシンプルな代表エラー画面を表示する
- 少なくとも `未対応形式`、`読み込み失敗`、`参照解決失敗` は区別する
- ユーザー向け表示は簡潔にし、必要なら詳細は別途見られる余地を残す

## 16. 開発フェーズ

### Phase 1

`Tauri + React + Three.js` で、MVP を成立させる。

到達目標:

- モデルを開ける
- テクスチャを開ける
- カメラ操作できる
- 最低限の表示切替がある
- 最近開いたファイルが使える

### Phase 2

重い処理と実務上の痛点を Rust 側で補強する。

候補:

- パス解決改善
- DDS 補助
- EXR 軽量化
- キャッシュ
- サムネイル

### Phase 3

必要なら描画やデコードの一部をネイティブ寄りに強化する。

候補:

- `wgpu` ベースの一部置換
- より高度なメモリ管理
- プレビュー性能改善

## 17. 現時点の仮決め

### 仮決め 1

製品カテゴリは「3D ビューア」ではなく「CG Asset Quick Look」とする。

### 仮決め 2

MVP は単一ファイルを速く快適に開くことを最優先にする。

### 仮決め 3

USD は最初から完全対応を約束しない。

### 仮決め 4

実装の初速を優先し、初期描画基盤は `Three.js` を使う。

### 仮決め 5

DDS は MVP に含めるが、初期品質は preview レベルを許容する。

### 仮決め 6

重い処理は必要になってから Rust 側へ逃がす。

### 仮決め 7

Windows 連携は MVP でファイル関連付けを含めるが、右クリック動線は含めない。

### 仮決め 8

フォルダ内前後移動は MVP に含め、同一フォルダ内の対応ファイル全体を順送りする。

### 仮決め 9

UI は `macOS Preview / Quick Look` と `XnView` を参照し、表示面積優先の最小構成にする。

### 仮決め 10

3D の追加 UI は、ワイヤーフレーム、環境マップ、背景切替、アニメーション再生系に絞る。

### 仮決め 11

3D カメラ操作は一般的な DCC に近い挙動を目指し、初期は `OrbitControls` 系をベースに成立させる。

### 仮決め 12

マテリアルは `Three.js` の標準 PBR を基準にし、完全再現ではなく preview に十分な再現性を優先する。

### 仮決め 13

初期カメラは `fit to view` と斜め上視点を基本にし、極端なスケールには警告を出す。

### 仮決め 14

ビューアの基準 up-axis は `Y-up` とする。

### 仮決め 15

背景は灰色ベース、環境マップは 1 個固定から始める。

### 仮決め 16

テクスチャ単体ビューは `RGB / RGBA / Alpha` 切り替えと `HDR / EXR` のトーンマップ調整を持つ。

### 仮決め 17

関連付けは MVP に含めるが、対象は代表的な 3D / CG アセット拡張子に絞り、一般画像拡張子は対象外にする。

### 仮決め 18

MVP ではキャッシュを持たず、速度は起動時間・初回表示時間・前後移動時間を実測で詰める。

### 仮決め 19

読み込み失敗時はシンプルな代表エラー画面を出し、未対応形式・読み込み失敗・参照解決失敗を区別する。

### 仮決め 20

メタデータ表示は MVP に含め、階層構造表示は早期対応候補として優先度高で扱う。

### 仮決め 21

モデルが参照するテクスチャの一覧と個別プレビューは、早期対応候補として優先度高で扱う。

### 仮決め 22

前後移動ショートカットは `Left Arrow / Right Arrow` を採用する。

### 仮決め 23

アニメーションバーは画面下部オーバーレイとし、アニメーション付きモデルでのみ表示する。

### 仮決め 24

カメラ操作は Maya 系を優先し、`Alt + 左 / 中 / 右ドラッグ` を基本にする。

### 仮決め 25

関連付け対象は 3D / CG アセット系対応拡張子を広めに候補化し、一般画像拡張子は除外する。

### 仮決め 26

`DDS` は可能な限り広く対応したいが、初期段階ではローダー実装が扱える範囲を保証ラインとする。

### 仮決め 27

最近開いたファイルは軽量なローカル JSON 設定で管理する。

### 仮決め 28

エラー表示は画面上は簡潔にしつつ、内部では未対応形式・読み込み失敗・参照解決失敗・テクスチャ欠損・スケール警告を区別する。

### 仮決め 29

関連付け拡張子は現行候補を広めに採用し、`mtl` は単体関連付けしない。`Alembic` は将来対応候補として扱う。

### 仮決め 30

初期のマテリアル確認モードは、`テクスチャなし`、`テクスチャあり`、`ワイヤーフレーム`、`テクスチャあり + ワイヤーフレーム` の 4 パターンとする。

### 仮決め 31

`DDS` の初期保証範囲は、実装負荷が過剰にならない範囲でローダーが安定して扱えるものに限定する。
